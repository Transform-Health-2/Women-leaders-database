Known Limitations & Strategic Decisions

Challenges addressed through deliberate design choices

🔗 LinkedIn — Form-first approach, not data scraping

LinkedIn has progressively locked down its API (removed 2015–2022). Profile scraping is now explicitly prohibited in their ToS and carries legal risk (hiQ v. LinkedIn case unresolved). Our solution: treat LinkedIn as a vanity identifier and hyperlink only. Leaders fill in their own structured data through our consent-backed form.

Why this is better: (1) Consent-backed — no GDPR exposure. (2) Structured — we get expertise tags, experience years, and countries of operation; LinkedIn bios are free-text. (3) Platform-independent — if LinkedIn changes, our data is ours. (4) Self-service updates — leaders update via magic link, no LinkedIn required.

Future: We may layer ORCID (open researcher identity) or Open Badges as verifiable credentials in v2 — both are open standards LinkedIn cannot block.

📸 Photo storage — RESOLVED via Supabase

✓ Mandatory photo upload in submit flow. Client-side compression. Supabase Storage handles file storage; Supabase Auth gates admin approval before URLs go public. Free tier sufficient (1GB storage, 2GB bandwidth).

Data Flow & User Journeys

How the system works end-to-end

Flow

The Complete Data Flow

① Submit

Leader submits profile (self or nominated) → Supabase leaders table (status: pending) + photo upload to Storage

↓

② Admin Review

Approve (status: live) → appears in public Directory. Reject (status: rejected) → removed from public.

↓

③ Leader Updates Profile

Leader requests update/delete → requests table (pending) → Admin approves with magic link (for updates) or confirms delete

↓

④ Analytics

Public users interact with Directory (search, filter, sort) and Analytics (region selection, specialisation filters). LinkedIn clicks tracked in real-time.

User Roles

👤 Public Visitor

Directory: Search/filter/sort leader cards. Click cards to open 1200px modal with bio, expertise, years of experience, LinkedIn link (click tracked). Analytics: Interact with world map, select regions, filter by specialisation. No login required. Read-only.

📝 Leader (Self-Submit)

Fill 5-step form: Branch → Consent → Basic Info (name, email, photo, country) → Profile Details (years, expertise, geo scope, countries of operation, bio) → Links (LinkedIn, achievements). Submit directly to Supabase. Status set to pending. See thank-you screen.

🎤 Nominator

Pick "Nominating someone else" → enter own name/email + nominee's name + LinkedIn URL. Submit directly to Supabase (branch: nominate). Admin sees in Nominated tab and sends outreach message.

🔐 Leader (Managing Profile)

Click "Manage or remove your profile" (on Submit page or in ProfileModal). Identify via name/email. Choose Update (describe change) or Delete (optional reason). Request saved to Supabase. Admin handles via magic link (updates) or manual confirm (deletes).

👨‍💼 Admin

Pending Submissions: Review, approve/reject, flag duplicates. Nominated: Send outreach, approve/reject. Profile Requests: Send magic links for updates, approve deletes. All Entries: Search/filter/sort all records, view LinkedIn clicks. Test Results: Live QA dashboard. User Manual: Complete workflow guide.

Where we are and why this is the right move

We built fast with Google Sheets + Apps Script and got 81 leaders into a working directory. Then we hit a wall: Firebase Storage never got off the ground, Apps Script requires a manual redeploy every time we change a line of backend code, there's no real admin login, and the magic link system is entirely hand-rolled. We are not migrating from a complete stack — we are completing it for the first time.

Supabase is the single platform that closes every open gap.

Admin login button is literally disabled — no real auth exists Supabase Auth gives proper email/password login in <1 hour

Identity strategy — beyond LinkedIn

🔭 Forward-thinking approach to professional identity

LinkedIn's API restrictions are getting tighter, not looser. Since 2022 they have removed profile access from most partners, blocked the majority of OAuth scopes, and updated their ToS to explicitly ban automated data collection. Treating LinkedIn as anything more than a hyperlink is a liability.

Our current position is already the right one. LinkedIn URL = a public link to verify who someone is. Nothing more. All structured data lives in our own database, submitted with explicit consent.

What we can build toward:

  • ORCID integration. ORCID is an open, non-profit researcher identity standard used by 20M+ researchers globally. It has a real API, no scraping needed, and is explicitly designed for exactly this kind of directory. Worth exploring as an optional "verify via ORCID" button for leaders with academic or research backgrounds.
  • Open Badges / Verifiable Credentials. Emerging W3C standard. Leaders could carry a "Transform Health Verified" credential in a digital wallet, portable across platforms — not locked to LinkedIn at all.
  • Email-as-identity. The magic link pattern we already have is effectively passwordless, self-sovereign identity. When Supabase Auth handles this, it becomes production-grade with no extra work.
  • Our data is our moat. A database of 81 (and growing) consented, structured, verified leader profiles is something LinkedIn cannot replicate — it is curated, purposeful, and consent-backed. That is the asset. LinkedIn is just a link.

Summary: Supabase migration + email OTP auth puts us in a position where professional identity is handled cleanly, without any dependency on platforms we do not control.
