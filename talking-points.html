Open challenges

🔗 LinkedIn URL scraping — why it won't work

The current Manage Profile flow uses a LinkedIn URL as an identity token to look up who is requesting a profile change. The intent was to also auto-fill profile data by scraping the LinkedIn page — saving leaders from re-entering their role, organisation, and photo.

The hard reality: LinkedIn aggressively blocks all automated scraping. Their terms of service prohibit it, they serve login walls to headless browsers, they rotate bot-detection fingerprinting, and they have taken legal action against scrapers (hiQ v. LinkedIn, 2022). Any scraper breaks within days of being built and puts the project at legal risk.

Practical alternatives: (1) Ask users to paste their LinkedIn URL as a vanity identifier only — no scraping, just a hyperlink. (2) Use the LinkedIn API (requires partner approval, significant overhead). (3) Let users fill the form themselves — which is what the current Submit flow already does well.

⚠️ Firebase Storage for photo uploads — proceed with caution

Firebase Storage is already wired up and working for photo uploads in the Submit flow. It's convenient, but there are real risks to understand before going to production.

Cost exposure
Firebase Storage bills per GB stored and per GB downloaded. If the directory grows to hundreds of profiles and gets traffic, costs can spike unexpectedly — especially with no CDN caching layer in front.

No moderation gate
Anyone who submits the form can upload any image. Without an admin review step, inappropriate or oversized files go straight into the bucket and are publicly readable.

Security rules
The default Firebase Storage rules are often left open during dev. Before launch, rules must restrict writes to authenticated users only and enforce file-type and size limits.

Recommendation
Keep Firebase for now but: (1) add file-type + 2 MB size validation client-side (compression util exists), (2) add admin approval before a photo URL is written to the profile record, (3) set a monthly budget alert in the Firebase console.

User Journey

Public

Public Visitor

Lands on the site and sees the Directory tab — a searchable, filterable grid of leader cards. They can:

  • Search by name, role, or org
  • Filter by continent, country, or expertise via dropdowns
  • Sort by Latest / A–Z / Expertise
  • Load more profiles (pagination)
  • Click a card to open a Profile Modal showing full details: role, org, country, bio, expertise, years of experience, LinkedIn link
  • Switch to the Analytics tab — world map, region tabs with leader counts, specialisation bar chart, click a region to see filtered leader cards below the map

No login, no account. Fully public and read-only.

Self Submit

Leader — Self Submit

  • Hits Submit Profile tab → sees two cards: "I am nominating myself" vs "Nominating someone else" — picks myself
  • Consent gate — must click "Yes I consent" to proceed. Declining ends the flow
  • Basic Info — first name, last name, email (private, not public), profile photo (MANDATORY, 5MB cap, compressed), country of residence, org, role
  • Profile Details — years of experience (pill selector), up to 5 expertise tags, "Other" with free-text, countries of operation (multi-select), bio (300–500 chars enforced)
  • Links & Achievements — LinkedIn URL, up to 3 notable items (title, link, type)
  • Hits Submit Profile → photo uploads to Firebase Storage, full payload POSTs to Apps Script → row written to Submissions sheet with status: pending
  • Sees a thank-you screen: "profile under review, 3–5 business days"
Nominator

Nominator — Submitting Someone Else

  • Same Step 0 but picks "I am nominating someone else"
  • Enters the nominee's LinkedIn or public profile URL
  • Hits Submit Nomination → goes directly to submit, skipping consent/bio/photo steps
  • A minimal row lands on the Sheet: branch: "nominate", nominate_link: <url>, everything else blank
  • Admin sees this in the Nominated tab and uses the pre-built outreach message to contact that person
Manage

Leader — Managing Their Profile

Already in the directory and wants to update or remove their listing:

  • Clicks "Manage or remove your profile" on the Submit page (or the button in their Profile Modal)
  • Identify step — enters first name, last name, email, optional LinkedIn
  • Hits "Find my profile" → calls the live Sheet, matches by exact first + last name, shows a "Profile found" card with their role and org
  • Two paths:
    • Update path: Picks "Update profile" → free-text box describing what to change ("new role: X, update bio to: …") → Reviews request → submits → profileRequest saved to Requests sheet, status pending
    • Delete path: Picks "Remove profile" → optional reason field → submits → same profileRequest row, request_type: delete
  • Magic link alternative (faster):
    • On the identify step, after profile is found, there's also "Or send my profile to this email instead →"
    • Triggers sendProfileLink → Apps Script looks up the profile, generates a one-time token, emails a magic link
    • User clicks link → token validated, marked used → profile snapshot returned to the FE → user lands directly in an edit view pre-filled with their current data
Admin

Admin

Navigates to the Admin console (separate route, not in public nav):

  • Pending Submissions tab — all self-submitted profiles awaiting review. Searchable + filterable by country, expertise, sort order. Expand any row to see full bio, profile details, email (private), LinkedIn. Approve → sets status: live → appears in public directory. Reject → sets status: rejected.
  • Nominated tab — nomination-only submissions (just a LinkedIn URL). Expand to see nominee details and a pre-built outreach message. Copy message to clipboard → reach out manually. Approve or reject once the person has been contacted and confirmed.
  • Profile Requests tab — update and delete requests from existing leaders.
    • Updates sub-tab — leader has described what they want changed; admin clicks "Send update link via email" → magic link goes to their inbox for self-service editing
    • Deletes sub-tab — leader wants to be removed; admin checks boxes and approves → profile set to rejected and removed from public directory
  • All Entries tab — full table of every record (live, pending, rejected) with search, sort, pagination, expandable detail rows.
  • Summary stats (Pending / Live / Rejected counts) always visible at the top.
Flow

The Full Data Flow

Submit form → Firebase (photo) + Apps Script (profile row, status: pending)
               ↓
               Admin reviews
                         ↓
         Approve → status: live → appears in Directory
           Reject → status: rejected → gone from public view
                         ↓
         Leader requests update → Requests sheet (pending)
                         ↓
         Admin sends magic link → one-time token emailed
                         ↓
         Leader clicks link → edits profile → new pending submission
                         ↓
         Admin approves again → goes live
